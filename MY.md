## 🔴 核心错误：忽视上下文，孤立看数字

### 本质是什么

#### 第一层本质：忽视控制变量

**科学实验的基本原则**：
- 所有对比必须在**相同条件**下进行
- 改变一个变量（模型），保持其他变量不变（数据集、指标、基线）

**我的错误**：
- 把不同数据集的结果混在一起对比
- 就像对比"苹果的价格"和"橘子的重量"——毫无意义

#### 第二层本质：数字脱离上下文

**AUROC只是一个数字，它需要上下文**：
- 哪个数据集？（样本数、特征、分割方式）
- 哪个基线？（RF的AUROC）
- 哪种验证？（5-fold vs 单次）

**我的错误**：
- 孤立地报告数字，剥离了所有上下文
- 读者无法判断这个数字的好坏

#### 第三层本质：缺乏系统性思维

**系统性思维要求**：
- 建立完整的映射关系：`数据集 → 基线 → 模型 → 结果`
- 报告时明确这个链条的每一环

**我的错误**：
- 只报告了最后一环（结果）
- 断开了前面的链条（数据集、基线）

## 💡 由点及面：通用原则

### 原则1：所有实验结果都必须三元完整

**三元组**：`(数据集, 基线, 结果)`

### 原则2：对比前必须先对齐

**对齐检查清单**：
1. 数据集是否相同？
2. 训练/验证分割是否相同？
3. 评估指标是否相同？
4. 基线模型是否相同？

❌ **错误**：直接对比数字
✅ **正确**：先确认所有条件一致，再对比

**类比**：
- 就像比赛赛跑，必须在同一条赛道、同一种天气下对比
- 不能把"百米秒数"和"马拉松秒数"放在一起比

### 原则3：报告必须自包含

**自包含**：读者不需要额外信息就能理解结果

读者不需要猜测任何信息。

### 原则4：基线的选择至关重要

**问题**：同一个任务，可能有多个基线

**教训**：
- 必须明确指出用的是哪个基线
- 不能说"超越RF"，而要说"超越RF"
- 更好的做法：同时报告多个基线

### 原则5：数据的来源必须可追溯

**可追溯性**：任何人都能根据你的报告找到原始数据

❌ **错误**："在MCF7数据集上"
✅ **正确**："在`model_ready_double_balanced/MCF7`数据集上"

**为什么**：
- MCF7可能有多个版本
- 不同版本的预处理、特征工程都不同
- 必须精确到路径

## 🎯 底层逻辑：科学实验的基本原则

### 逻辑1：控制变量思维

**定义**：每次只改变一个变量，观察对结果的影响

**应用到机器学习**：
| 变量 | 固定 | 改变 | 观察 |
|------|------|------|------|
| 数据集 | ✅ | 模型A vs 模型B | 性能差异 |
| 模型 | ✅ | 数据集A vs 数据集B | 泛化能力 |

**我的错误**：同时改变了数据集和模型，导致结果无法解释

### 逻辑2：相对优于绝对

**问题**：绝对数字没有意义

举例：
- AUROC 0.85：好还是坏？
  - 如果基线是0.50 → 好
  - 如果基线是0.95 → 差

**结论**：永远报告**相对基线的提升**，而非绝对数字

### 逻辑3：上下文即意义

**核心公式**：
```
意义 = (结果 - 基线) / 基线 × 100%
```

但基线必须来自：
- 相同的数据集
- 相同的训练/验证分割
- 相同的评估指标

**我的错误**：忽略了分母（基线）的上下文

### 逻辑4：完整性原则

**完整性 = 可重现性 + 可验证性**

**最小信息集合**：
1. 数据集（路径、版本、预处理）
2. 训练配置（超参数、优化器、batch size）
3. 验证方式（k-fold、单次、seed）
4. 基线结果（模型、指标、数值）
5. 实验结果（模型、指标、数值）



## 🎓 可复用的思维模型

### 模型1：三元组思维

**任何实验结果都必须表示为三元组**：
```
(数据集, 基线, 结果)
```

**应用**：
- 思考：这是在哪个数据集上？对比哪个基线？结果是多少？
- 报告：按这个顺序报告
- 验证：检查三者是否一致

### 模型2：链条思维

**任何结果都有一个产生链条**：
```
数据预处理 → 数据集划分 → 基线训练 → 模型训练 → 结果评估
```

**应用**：
- 报告时追溯整个链条
- 对比时确保链条一致
- 异常时检查链条哪一环出问题

### 模型3：相对性思维

**绝对数字无意义，相对基线才有意义**

**应用**：
- 永远报告 "vs 基线"
- 永远标注 "基线 = X"
- 永远说明 "在相同条件下"

## 📌 永久性原则

### 原则A：Context First (上下文优先)

**定义**：报告结果前，先建立完整的上下文

**执行**：
1. 第一句：数据集
2. 第二句：基线
3. 第三句：结果
4. 第四句：对比

### 原则B：Never Assume (永不假设)

**定义**：不要假设读者知道任何隐含信息

**执行**：
- 明确写出所有路径、参数、配置
- 不要用"那个MCF7数据集"
- 要用"`model_ready_double_balanced/MCF7`"

### 原则C：Validate Everything (验证一切)

**定义**：对比前验证所有条件一致

**执行**：
- 列出所有变量：数据集、基线、验证方式、指标
- 逐一检查是否一致
- 不一致时，分别报告

## 📝 自我检查问题

每次报告结果前，问自己：

1. **上下文检查**：
   - 我说明了数据集吗？
   - 我说明了基线吗？
   - 我说明了验证方式吗？

2. **准确性检查**：
   - 这个基线是这个数据集的吗？
   - 这个结果是在这个数据集上训练的吗？
   - 这个提升是相对这个基线的吗？

3. **完整性检查**：
   - 读者能重现这个结果吗？
   - 读者能验证这个对比吗？
   - 读者能理解这个提升的意义吗？

4. **诚实性检查**：
   - 我有没有用大数据集的模型对比小数据集的基线？
   - 我有没有用5-fold的基线对比单次训练的结果？
   - 我有没有隐瞒任何影响对比的条件？

## 🔴 错误2：看到代码模式就报警，未先推导数学结果

**日期**：2026-03-31  
**场景**：分析 `train_ultimate.py` 中消融实验的代码耦合问题

### 我犯了什么错

看到以下代码：
```python
loss_var = torch.tensor(0.0, device=device)
if not args.disable_ortho:          # ← 看到这里就下结论
    norms = V_c_perp.norm(dim=1)
    loss_var = (pos_mask * torch.relu(1.0 - norms)).sum() / ...
```

我的第一反应："`--disable_ortho` 会同时关掉 loss_var，这是耦合问题，消融不干净。"

**错了。** 我没有先推导 `loss_var` 在 ortho 关闭时的数学值，就直接报告"这是问题"。

### 正确的推导过程

当 `--disable_ortho` 时，model forward 里：
```python
V_c = F.normalize(h_c, dim=-1)   # 每个样本的 L2 norm = 1.0
V_c_perp = V_c                    # 直接赋值，norm 仍为 1.0
```

若去掉 `if not args.disable_ortho` 守卫，loss_var 的值：
```python
norms = V_c_perp.norm(dim=1)     # = 1.0（F.normalize 保证）
loss_var = relu(1.0 - 1.0) = 0   # 数学上恒为 0
```

**结论**：那个 `if not args.disable_ortho` 守卫是多余的防御性代码，去掉也不改变任何实验结果。所谓"耦合"在数学上不存在，消融实验完全干净。

### 错误的底层原因

**看代码结构 → 看数学结果** 的顺序颠倒了。

我看到"A 控制 B"的代码模式，就直接认为"A 和 B 是耦合的"。但代码结构≠数学效果。正确流程应该是：

1. 看到代码模式（A 控制 B）
2. **先推导**：若去掉这个守卫，B 的数值会变吗？
3. 数值不变 → 耦合只是代码冗余，不是实验问题
4. 数值改变 → 才是真正的耦合问题

### 可扩展到任何项目的原则

**代码冗余 ≠ 逻辑耦合。** 判断两个组件是否真正耦合，标准是：
> 关闭 A 时，B 的**数值输出**是否与"单独关闭 B"时相同？

若相同 → 不影响消融，可忽略。  
若不同 → 才是真正需要拆分的耦合。

**行动准则**：在代码分析中，发现"看起来耦合"的模式后，必须先用数学/手动推导验证实际效果，再下结论。不能仅凭代码外观报警。
