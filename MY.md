## 🔴 核心错误：忽视上下文，孤立看数字

### 本质是什么

#### 第一层本质：忽视控制变量

**科学实验的基本原则**：
- 所有对比必须在**相同条件**下进行
- 改变一个变量（模型），保持其他变量不变（数据集、指标、基线）

**我的错误**：
- 把不同数据集的结果混在一起对比
- 就像对比"苹果的价格"和"橘子的重量"——毫无意义

#### 第二层本质：数字脱离上下文

**AUROC只是一个数字，它需要上下文**：
- 哪个数据集？（样本数、特征、分割方式）
- 哪个基线？（RF的AUROC）
- 哪种验证？（5-fold vs 单次）

**我的错误**：
- 孤立地报告数字，剥离了所有上下文
- 读者无法判断这个数字的好坏

#### 第三层本质：缺乏系统性思维

**系统性思维要求**：
- 建立完整的映射关系：`数据集 → 基线 → 模型 → 结果`
- 报告时明确这个链条的每一环

**我的错误**：
- 只报告了最后一环（结果）
- 断开了前面的链条（数据集、基线）

## 💡 由点及面：通用原则

### 原则1：所有实验结果都必须三元完整

**三元组**：`(数据集, 基线, 结果)`

### 原则2：对比前必须先对齐

**对齐检查清单**：
1. 数据集是否相同？
2. 训练/验证分割是否相同？
3. 评估指标是否相同？
4. 基线模型是否相同？

❌ **错误**：直接对比数字
✅ **正确**：先确认所有条件一致，再对比

**类比**：
- 就像比赛赛跑，必须在同一条赛道、同一种天气下对比
- 不能把"百米秒数"和"马拉松秒数"放在一起比

### 原则3：报告必须自包含

**自包含**：读者不需要额外信息就能理解结果

读者不需要猜测任何信息。

### 原则4：基线的选择至关重要

**问题**：同一个任务，可能有多个基线

**教训**：
- 必须明确指出用的是哪个基线
- 不能说"超越RF"，而要说"超越RF"
- 更好的做法：同时报告多个基线

### 原则5：数据的来源必须可追溯

**可追溯性**：任何人都能根据你的报告找到原始数据

❌ **错误**："在MCF7数据集上"
✅ **正确**："在`model_ready_double_balanced/MCF7`数据集上"

**为什么**：
- MCF7可能有多个版本
- 不同版本的预处理、特征工程都不同
- 必须精确到路径

## 🎯 底层逻辑：科学实验的基本原则

### 逻辑1：控制变量思维

**定义**：每次只改变一个变量，观察对结果的影响

**应用到机器学习**：
| 变量 | 固定 | 改变 | 观察 |
|------|------|------|------|
| 数据集 | ✅ | 模型A vs 模型B | 性能差异 |
| 模型 | ✅ | 数据集A vs 数据集B | 泛化能力 |

**我的错误**：同时改变了数据集和模型，导致结果无法解释

### 逻辑2：相对优于绝对

**问题**：绝对数字没有意义

举例：
- AUROC 0.85：好还是坏？
  - 如果基线是0.50 → 好
  - 如果基线是0.95 → 差

**结论**：永远报告**相对基线的提升**，而非绝对数字

### 逻辑3：上下文即意义

**核心公式**：
```
意义 = (结果 - 基线) / 基线 × 100%
```

但基线必须来自：
- 相同的数据集
- 相同的训练/验证分割
- 相同的评估指标

**我的错误**：忽略了分母（基线）的上下文

### 逻辑4：完整性原则

**完整性 = 可重现性 + 可验证性**

**最小信息集合**：
1. 数据集（路径、版本、预处理）
2. 训练配置（超参数、优化器、batch size）
3. 验证方式（k-fold、单次、seed）
4. 基线结果（模型、指标、数值）
5. 实验结果（模型、指标、数值）



## 🎓 可复用的思维模型

### 模型1：三元组思维

**任何实验结果都必须表示为三元组**：
```
(数据集, 基线, 结果)
```

**应用**：
- 思考：这是在哪个数据集上？对比哪个基线？结果是多少？
- 报告：按这个顺序报告
- 验证：检查三者是否一致

### 模型2：链条思维

**任何结果都有一个产生链条**：
```
数据预处理 → 数据集划分 → 基线训练 → 模型训练 → 结果评估
```

**应用**：
- 报告时追溯整个链条
- 对比时确保链条一致
- 异常时检查链条哪一环出问题

### 模型3：相对性思维

**绝对数字无意义，相对基线才有意义**

**应用**：
- 永远报告 "vs 基线"
- 永远标注 "基线 = X"
- 永远说明 "在相同条件下"

## 📌 永久性原则

### 原则A：Context First (上下文优先)

**定义**：报告结果前，先建立完整的上下文

**执行**：
1. 第一句：数据集
2. 第二句：基线
3. 第三句：结果
4. 第四句：对比

### 原则B：Never Assume (永不假设)

**定义**：不要假设读者知道任何隐含信息

**执行**：
- 明确写出所有路径、参数、配置
- 不要用"那个MCF7数据集"
- 要用"`model_ready_double_balanced/MCF7`"

### 原则C：Validate Everything (验证一切)

**定义**：对比前验证所有条件一致

**执行**：
- 列出所有变量：数据集、基线、验证方式、指标
- 逐一检查是否一致
- 不一致时，分别报告

## 📝 自我检查问题

每次报告结果前，问自己：

1. **上下文检查**：
   - 我说明了数据集吗？
   - 我说明了基线吗？
   - 我说明了验证方式吗？

2. **准确性检查**：
   - 这个基线是这个数据集的吗？
   - 这个结果是在这个数据集上训练的吗？
   - 这个提升是相对这个基线的吗？

3. **完整性检查**：
   - 读者能重现这个结果吗？
   - 读者能验证这个对比吗？
   - 读者能理解这个提升的意义吗？

4. **诚实性检查**：
   - 我有没有用大数据集的模型对比小数据集的基线？
   - 我有没有用5-fold的基线对比单次训练的结果？
   - 我有没有隐瞒任何影响对比的条件？

## 🔴 错误2：看到代码模式就报警，未先推导数学结果

**日期**：2026-03-31  
**场景**：分析 `train_ultimate.py` 中消融实验的代码耦合问题

### 我犯了什么错

看到以下代码：
```python
loss_var = torch.tensor(0.0, device=device)
if not args.disable_ortho:          # ← 看到这里就下结论
    norms = V_c_perp.norm(dim=1)
    loss_var = (pos_mask * torch.relu(1.0 - norms)).sum() / ...
```

我的第一反应："`--disable_ortho` 会同时关掉 loss_var，这是耦合问题，消融不干净。"

**错了。** 我没有先推导 `loss_var` 在 ortho 关闭时的数学值，就直接报告"这是问题"。

### 正确的推导过程

当 `--disable_ortho` 时，model forward 里：
```python
V_c = F.normalize(h_c, dim=-1)   # 每个样本的 L2 norm = 1.0
V_c_perp = V_c                    # 直接赋值，norm 仍为 1.0
```

若去掉 `if not args.disable_ortho` 守卫，loss_var 的值：
```python
norms = V_c_perp.norm(dim=1)     # = 1.0（F.normalize 保证）
loss_var = relu(1.0 - 1.0) = 0   # 数学上恒为 0
```

**结论**：那个 `if not args.disable_ortho` 守卫是多余的防御性代码，去掉也不改变任何实验结果。所谓"耦合"在数学上不存在，消融实验完全干净。

### 错误的底层原因

**看代码结构 → 看数学结果** 的顺序颠倒了。

我看到"A 控制 B"的代码模式，就直接认为"A 和 B 是耦合的"。但代码结构≠数学效果。正确流程应该是：

1. 看到代码模式（A 控制 B）
2. **先推导**：若去掉这个守卫，B 的数值会变吗？
3. 数值不变 → 耦合只是代码冗余，不是实验问题
4. 数值改变 → 才是真正的耦合问题

### 可扩展到任何项目的原则

**代码冗余 ≠ 逻辑耦合。** 判断两个组件是否真正耦合，标准是：
> 关闭 A 时，B 的**数值输出**是否与"单独关闭 B"时相同？

若相同 → 不影响消融，可忽略。  
若不同 → 才是真正需要拆分的耦合。

**行动准则**：在代码分析中，发现"看起来耦合"的模式后，必须先用数学/手动推导验证实际效果，再下结论。不能仅凭代码外观报警。

## 📌 项目基本约束与科学目标（勿混淆）

**更新日期**：2026-04-17

### 任务定义（不可改变）

- **输入**：药物 SMILES + 基因 DNA 序列（仅此两者）
- **输出**：二分类——显著 CGI 的方向（上调 vs 下调），仅对 |z|>2 的显著 pair 训练和评估
- **不做回归**：任务是方向分类，z-score 连续值不作为训练目标
- **无预训练模型**：所有编码器端到端从头训练，追求端到端可解释性
- **无外部生物数据库**：不用 PPI 网络、基因表达谱、GO 标注等作为输入特征

### 多细胞系的科学问题（不是性能优化）

**多细胞系 ≠ 联合训练**。每个细胞系独立训练是有意为之。

科学问题是：**相同的 (drug SMILES, gene sequence) 输入，为什么在不同细胞系中预测结果不同？**

这个差异本身就是研究对象——细胞系异质性（cell-line heterogeneity）。联合训练会混淆这个信号，使模型学到"平均行为"而非"各细胞系独特行为"。

正确分析路径：
1. 各细胞系独立训练 → 得到各自的谱模式
2. 跨细胞系比较谱模式 → 找到共有模式（药物机制保守）和细胞系特异模式（异质性来源）

### 算子秩 r 的结论（已实验验证）

r=8/16/32 均已测试，AUC 无显著差异。**r 不是性能瓶颈**，不再尝试改变 r。

### 当前性能瓶颈的正确诊断

给定约束（仅 SMILES + DNA 序列），模型的真正瓶颈在于：
1. **基因编码器**：DNA k-mer 描述的是序列统计特征，不是基因功能。同一基因在不同细胞系中的调控状态无法从序列中读出。这是任务本身的信息论上限。
2. **药物编码器**：当前 GIN 使用基本原子特征，缺少键级别特征（键型、芳香性、手性）。这是可改进的。

**核心教训**：在"仅序列输入"的约束下，模型性能的天花板由序列信息量决定。提升性能的正确路径是**提升编码器的信息提取效率**，而非增加融合路径或改变任务定义。

## 💡 架构创新：SBB + Tanimoto（2026-04-17）

### 创新一：SharedBiologicalBasis (SBB)

**文件**：`New/train_sbb.py`，开关：`--use_sbb`

**问题**：GeneMultiHeadReader 的 r 个 attn_queries 与 PharmacophoreExtractor 的 r 个 slot queries 完全独立，两套空间只靠算子层的 `⟨V_j, h_g_modes_j⟩` 事后拉齐。算子同时承担"学扰动"和"对齐两个空间"两个任务，负担过重。

**解法**：共享一套可学习基 `B ∈ R^{r×H}`，两个编码器都用 `B[j]` 作为第 j 个查询向量。药物和基因天然在同一生物学坐标系下被描述，算子只需学强度 σ。

**第一性原理**：空间对齐应当是架构的归纳偏置，而非由数据从头学习。将"事后对齐"提前为"事前统一"，在相同参数量下（实际还少 1024 参数）提供更强的结构约束。

### 创新二：Tanimoto 谱一致性损失

**文件**：`New/train_sbb.py`，开关：`--lam_tanimoto X`

**问题**：SpectrumDirectionCL 失败因为方向信号被 BCE 完全覆盖（冗余监督）。需要一个 BCE 看不到的新信号。

**解法**：结构相似的药物应激活相似的生物学程序。用 RDKit 计算 Morgan FP（2048-bit，无预训练，纯算法），batch 内对所有药物对计算 `Tanimoto_ij` vs `cosine(σ_i, σ_j)` 的 MSE。

**第一性原理**：`loss_tanimoto = MSE(cosine(σ_i, σ_j), Tanimoto(drug_i, drug_j))`。药物的结构关系是独立于 (drug, gene) 标签的新信息源，直接约束谱空间的几何结构与化学空间一致，对化学冷分割泛化有直接帮助。

**区别于 CL 失败的原因**：CL 监督的是每对 (drug, gene) 的方向（BCE 已覆盖）；Tanimoto 监督的是 drug-drug 的结构关系（BCE 完全看不到）。

### 消融实验设计

| 配置 | 命令标志 | 对比目的 |
|------|---------|---------|
| base | （使用 train_operator_moe.py） | 原论文基准 |
| +SBB | `--use_sbb` | SBB 单独贡献 |
| +Tan | `--lam_tanimoto 0.1` | Tanimoto 单独贡献 |
| +Both | `--use_sbb --lam_tanimoto 0.1` | 两者协同效果 |

## 🔴 错误：将"信息融合"等同于"性能提升"

**犯错场景**：设计 HMSD 时，让 U、V 也接受 gene 输入（joint = cat([pharma,gene])），认为"三要素全双边 = 更强的互动 = 更好的结果"。

**实际结果**：HMSD AUC=0.8832，低于基准 0.8923（-0.0091）。

### 错误的核心逻辑

把"信息的物理叠加次数"与"模型学到的信息量"画了等号。实际上：

- gene 在 HMSD 里出现 4 次（joint→V，coupling=V·gene，joint→U，分类器 h_g_global），但这 4 次的信息源头都是同一个 GeneMultiHeadReader 的输出
- 同一个有限表示的多次自积，不会产生新信息，只会放大噪声并增加过拟合

**第一性原理**：
> 信息量 = 信息源的熵，不是信息流经路径的数量。  
> 在同一个低熵信息源上叠加更多连接，不会增加系统熵。

### 正确的思考框架

**问自己两个问题**：
1. 这个新连接/路径，引入的是**新信息**还是**同一信息的重复**？
2. 额外参数量 / 有效训练样本数的比例是上升还是下降？

cold split 场景下（~5K 样本），任何使参数量增大的设计都需要极强的归纳偏置来弥补，否则一律适得其反。

### 可扩展到任何项目的普适法则

> **信息流冗余定律**：将同一表示反复融合进模型的不同层，不能增加模型的信息量，只能增加过拟合风险和梯度传播复杂度。
> 
> **有效改进的方向**：增加独立信息源（新模态/新数据）或增加监督密度（对比学习/自监督），而非增加参数或融合路径。

## 🔴 错误：将"语义聚合单元"错误地定义为 pairs 而非 entities

**犯错场景**：`go_enrich.py` 中 interaction view 逻辑——取正例集中 spectrum 值最高的 top-100 **pairs** 做 GO 富集。

**实际代码（旧，有 bug）**：
```python
top_pair_idx = np.argsort(pos_spectrum_j)[-args.top_k:]
top_inter_gene_ids = pos_gene_ids[top_pair_idx]  # 同一基因可出现多次
```

**问题**：
- 单个基因可因被多个不同药物激活而反复出现在 top-100 pairs 中
- Enrichr 接收到的 gene set 虽然包含 ~95 unique genes（同等有效），但**选基因的语义已经错了**：挑选的是"至少有一个 drug-gene 对高 spectrum"的基因，而非"平均 spectrum 最高"的基因
- 两种方法的 top-100 unique genes 仅有 13-31% 重叠（Mode 6 仅 13/100 重叠）
- 错误结果使 Mode 5 NF-κB (FDR=2.13e-3) 被错误报告；修正后 Mode 5 不显著，Mode 6 cell-cycle (FDR=1.37e-3) 成为最强 mode

**正确做法（新）**：
```python
df_pos = pd.DataFrame({"gene_id": pos_gene_ids, "spec_j": pos_spectrum_j})
gene_mean_spec = df_pos.groupby("gene_id")["spec_j"].mean().sort_values(ascending=False)
top_inter_gene_ids = gene_mean_spec.head(args.top_k).index.values
```

**第一性原理**：
> GO 富集的语义是"哪些**实体**（基因）与该生物过程关联"，不是"哪些**关系**（pairs）值最高"。  
> 任何 entity-level 问题（哪些基因？哪些药物？），聚合统计量必须以 entity 为单位做 groupby 后再排序，不能以 pair 为单位直接排序后取 top。

**可扩展到任何项目的普适法则**：
> 在 entity-relationship 数据中（drug-gene pairs、user-item ratings、patient-event logs），如果你的问题是"哪些 entity 表现最好/最活跃"，答案必须先 groupby entity 再 aggregate，然后排序。直接排序 pair-level 分数会导致高频实体（在多个 pair 中出现的 entity）被系统性地过代表。这是数据聚合中最常见的单位错误（unit-of-analysis error）。

## 🔴 错误：忽视 ReLU 对值域的约束

**犯错场景**：Bilateral Sigma 实现时写了 `sigma_q = nn.Sequential(nn.Linear(H,d), nn.ReLU())`，导致 sigma 恒为非负，AUC 长期卡在 0.765。

**第一性原理**：
> 任何激活函数都隐式地约束了输出的值域。在设计计算路径时，必须先问：这个路径的输出应该允许哪些符号/值域？ReLU 截断负值 = 削减一半的表达空间。

**可扩展法则**：
> 设计注意力/相似度/强度分数时，若分数需要表达"排斥"（负值）而非仅"吸引"（正值），中间表示不能有 ReLU。用 tanh（有界，含负值）或无激活（无界）替代。
